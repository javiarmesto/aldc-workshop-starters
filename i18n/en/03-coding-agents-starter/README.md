## What this is

Starting template for **Block 03 of the ALDC V-Valley 2026 Workshop**.

In this block you will use **ALDC agents** (`al-architect` and the `al-spec.create` workflow) on a **real business case** — incident management for a coffee shop chain. The goal **is not to produce AL code yet** — that comes in Block 04. Here you stop at two artefacts:

1. **`architecture.md`** · generated by `al-architect`
2. **`spec.md`** · generated by the `al-spec.create` workflow

And you use the `MEDIUM-HIGH complexity` path with HITL between each phase.

## The block input · `docs/caso-barista-incidents.md`

Open it and read it. **It is the most important thing**.

It is an internal email — the kind of material you would start thinking about architecture from in a real project. It has detail, it has ambiguities, it has things the architect will have to decide:

- Which tables and with what relationships
- How to structure severity (enum vs table)
- How to model automatic assignment (matrix? rules?)
- Which events to publish
- Which APIs to expose (REST v2.0 with bound actions)
- How to partition permissions

**Note:** at the end of the file there is a section "Technical team notes" pointing to ADR candidates. These are hints to help you guide the architect, not the solution.

## How to use it during Block 03

### Step 0 · set up environment

Follow [`docs/INSTALACION-ALDC.md`](./docs/INSTALACION-ALDC.md) to get ALDC
installed in your workspace. It takes 2 minutes.

```bash
# Fork this repo into your GitHub account, then:
git clone https://github.com/<your-username>/workshop-b03-start.git
cd workshop-b03-start
code .

# Install ALDC: Ctrl+Shift+X → search "AL Development Collection"
# Then: Ctrl+Shift+P → "AL Collection: Install Toolkit to Workspace"
# Reload window: F1 → "Developer: Reload Window"
```

### Step 1 · invoke al-architect

Open Copilot Chat and paste:

```
Load al-architect.

Context:
Read the business context in docs/caso-barista-incidents.md.
This is a Business Central extension for a coffee shop chain managing
operational incidents. Follow .github/copilot-instructions.md for
naming, ID range, and pattern constraints.

Task: Produce barista-incidents.architecture.md at MEDIUM-HIGH complexity.
Include:
- Skills applied (at the top)
- ADRs (numbered, with alternatives considered and decision rationale)
- Data model (tables, relationships, cardinalities)
- APIs exposed (v2.0, bound actions where applicable, queries)
- Events published (integration events for state transitions)
- Permissions strategy (Basic and Full sets)

Do NOT write AL code. Architecture only.
```

**What to expect in the output**:

- First line of the file: `> **Skills applied**: skill-api, skill-events, skill-pages, skill-permissions` (or a similar set)
- 4-6 numbered ADRs with considered alternatives
- Diagram or list of data model entities
- APIs with their bound actions
- Published events for state transitions
- Two proposed permission sets

### Step 2 · review the architecture (HITL gate)

**Do not skip this step.** The point of ALDC is that a human signs off between phases.

Open the generated `architecture.md` and ask yourself three questions:

1. **Do the ADRs reflect the reality of the case?** · Marta's email says assignment must be "configurable without touching code" · if the ADR proposes a fixed `enum` for assignment, push back.

2. **Does the data model cover what was requested?** · the weekly report needs to aggregate by type, by store, by machine · are there enough fields?

3. **Are any events missing?** · the email mentions that "high severity triggers push to the technician" · is there an integration event for that?

If something does not add up, ask for refinement:

```
The ADR about severity modeling chose enum. The email explicitly says
severity thresholds should be configurable. Please reconsider — table or
enum + override table would be alternatives worth evaluating.
```

### Step 3 · invoke the al-spec.create workflow

Once the architecture is approved:

```
@workspace use al-spec.create

Source: barista-incidents.architecture.md
Output: barista-incidents.spec.md
```

The workflow will ask questions to expand the architecture into an executable spec. **Answer them.** Do not run it "to see what happens" · the workflow is dialogic by design.

**What to expect in the output**:

- Concrete list of objects with tentative IDs assigned within the range 50100-50199
- Fields of each table with type and constraints
- Dependencies between objects (creation order)
- Expected test cases (happy path + edge cases)
- Phase plan for the conductor · how many phases, what each one does

### Step 4 · stop here

**Do not invoke al-conductor in this block.** Running the full pipeline from the spec is the exercise of Block 04.

Save the two artefacts (`architecture.md` and `spec.md`) in `docs/` for Block 04, or somewhere you will find them easily.

## Optional exercises · for those running ahead

### Exercise 1 · compare two `architecture.md` outputs

Run `al-architect` **twice** with the same input, in separate chats. Compare:

- Does it choose the same Skills?
- Does it propose the same tables?
- Do the ADRs match?

This is a way to calibrate how much variability the model introduces. Spoiler: the structure is stable, the details vary.

### Exercise 2 · force LOW complexity

In a new chat:

```
Load al-architect.

Context: read docs/caso-barista-incidents.md. Take the SIMPLEST POSSIBLE
interpretation — assume only 1 incident type ("machine breakdown"), no
severity levels, no external integration, no role center tiles. Just CRUD.

Task: Produce the MINIMUM architecture at LOW complexity. Single table,
basic page, basic permission set.
```

Compare with the MEDIUM-HIGH output from the main exercise. This is a good illustration of how the same input produces very different solutions depending on the declared complexity.

### Exercise 3 · request explicit trade-offs

Go back to the main `architecture.md` and ask:

```
For the 3 most critical ADRs, explain the trade-offs we accept by taking
the decided path. What would break if we later had to reverse each decision?
```

Good practice — it leaves you a pain map for the future.

## When Block 04 comes

You will bring:

- Your `architecture.md` (approved by yourself in Step 2)
- Your `spec.md` (dialogically built in Step 3)

And you will use `al-conductor` + its 3 sub-agents (al-planning, al-implement, al-review) to **implement** the first phases of the spec. Block 04 is green-field + brown-field · two approaches to the same case.
